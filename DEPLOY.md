# Деплой MovieNest: пошаговая инструкция

Сайт состоит из двух частей, и им нужны два разных хостинга:
- **Frontend** (React/Vite) → GitHub Pages — только статика, бесплатно, уже настроено через `.github/workflows/deploy-pages.yml`.
- **Backend** (Express API) → GitHub Pages его не запустит (это не Node-сервер). Разворачиваем на Render (есть бесплатный план).

## Если сайт сейчас показывает README вместо самого сайта

Это значит, что в настройках репозитория GitHub Pages ещё указано «брать файлы из ветки» — а туда мы ничего не публикуем, там просто лежит исходный код, и GitHub сам рендерит `README.md` в html-страницу. **Это самый частый и самый важный шаг** — без него ничего из остального не заработает, даже если всё остальное настроено идеально.

**Settings → Pages → Build and deployment → Source → выбрать "GitHub Actions"** (сейчас там наверняка стоит "Deploy from a branch").

После этого зайдите во вкладку **Actions** репозитория — там должен появиться workflow "Deploy frontend to GitHub Pages". Если он не запустился сам, откройте его и нажмите **Run workflow**. Через 1–2 минуты в этой же вкладке появится зелёная галочка — и сайт обновится.

## Шаг 0. Создайте бесплатную базу данных MongoDB Atlas

Раньше пользователи, XP, лидерборд, достижения и комментарии хранились в локальных JSON-файлах внутри контейнера backend'а. Проблема: на бесплатном тарифе Render (как и на большинстве бесплатных хостингов) диск контейнера **временный** — при "засыпании"/пробуждении сервиса после 15 минут простоя и при каждом новом деплое файловая система пересоздаётся с нуля, и все данные, записанные после последнего деплоя, стираются. Именно поэтому XP "терялось" со временем и после повторного входа.

Теперь эти данные хранятся в MongoDB — отдельном хостинге базы данных, который не зависит от жизненного цикла контейнера backend'а. Настройка бесплатная и занимает 5–10 минут:

1. Зарегистрируйтесь на [mongodb.com/cloud/atlas/register](https://www.mongodb.com/cloud/atlas/register) (можно через Google-аккаунт).
2. Создайте новый проект, затем **Build a Database → M0 Free** (бесплатный кластер, 512 МБ — этого более чем достаточно для XP/пользователей/комментариев).
3. Выберите любой ближайший регион (провайдер AWS/GCP/Azure — не важно) и нажмите **Create**.
4. **Database Access** (в меню слева) → **Add New Database User**: придумайте имя пользователя и пароль (сохраните их, они понадобятся в строке подключения). Права — **Read and write to any database**.
5. **Network Access** (в меню слева) → **Add IP Address** → **Allow Access from Anywhere** (`0.0.0.0/0`). Это нужно, потому что Render не даёт фиксированный IP на бесплатном плане — сервер обращается к базе с разных адресов.
6. **Database → Connect → Drivers** → скопируйте строку подключения, она выглядит так:
   ```
   mongodb+srv://<username>:<password>@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority
   ```
   Замените `<username>` и `<password>` на реальные значения из шага 4, и добавьте имя базы данных перед знаком `?` — например:
   ```
   mongodb+srv://myuser:mypassword@cluster0.xxxxx.mongodb.net/movienest?retryWrites=true&w=majority
   ```
   (имя `movienest` можно выбрать любое — MongoDB создаст такую базу автоматически при первой записи).

Эту готовую строку подключения используйте как значение переменной **MONGODB_URI** на шаге 1 ниже.

### Если сайт уже работал раньше и в нём есть реальные пользователи

Если в `backend/data/users.json` (и соседних файлах) уже накопились реальные аккаунты, XP, история просмотров и т.п. — не начинайте "с чистого листа". Перенесите их одной командой:

```bash
cd backend
cp .env.example .env      # если ещё не сделано
# впишите в .env реальный MONGODB_URI из шага выше
npm install
npm run migrate
```

Скрипт (`backend/scripts/migrate-json-to-mongo.js`) один раз копирует все записи из старых JSON-файлов в MongoDB и ничего не удаляет — его безопасно запускать даже повторно.

## Шаг 1. Разверните backend на Render

1. Зарегистрируйтесь на [render.com](https://render.com) и подключите свой GitHub-аккаунт.
2. New → Blueprint → выберите ваш репозиторий. Render найдёт `render.yaml` в корне и сам предложит создать сервис `movienest-backend`.
3. Перед деплоем задайте переменные окружения **CLIENT_ORIGIN** и **MONGODB_URI**:
   ```
   CLIENT_ORIGIN = https://<ваш-username>.github.io
   ```
   (без пути `/<репозиторий>/` и без слэша на конце — CORS сверяет именно домен, путь не важен).
   ```
   MONGODB_URI = <строка подключения из Шага 0>
   ```
   `JWT_SECRET` Render сгенерирует сам, ничего вводить не нужно.
4. Дождитесь деплоя и скопируйте адрес сервиса — что-то вроде:
   ```
   https://movienest-backend.onrender.com
   ```

> Бесплатный план Render "засыпает" после 15 минут без запросов и просыпается ~30–60 секунд при первом запросе — это нормально для бесплатного тарифа, не баг.

## Шаг 2. Подключите фронтенд к этому backend

1. В репозитории на GitHub: **Settings → Secrets and variables → Actions → Variables → New repository variable**.
2. Имя: `VITE_API_ROOT`, значение:
   ```
   https://movienest-backend.onrender.com/api
   ```
   (используйте свой реальный адрес с Render, обязательно с `/api` на конце).

## Шаг 3. Включите GitHub Pages через Actions

Описано выше — **Settings → Pages → Source → GitHub Actions**, затем запустить workflow (пуш в `main` или вручную через **Actions → Run workflow**).

Путь до сайта (например `/MovieNest--/`) вычисляется автоматически шагом `configure-pages` в workflow — вручную его нигде вписывать не нужно, даже если вы переименуете репозиторий.

## Проверка

- Откройте сайт, попробуйте зарегистрироваться/войти — если куки настроены правильно, после входа страница профиля должна открываться без повторного логина.
- Если видите ошибку CORS в консоли браузера (F12 → Console) — проверьте, что `CLIENT_ORIGIN` на Render совпадает с адресом GitHub Pages *точно* (без слэша на конце).
- Если после входа профиль "слетает" — проверьте, что backend задеплоен с `NODE_ENV=production` (в `render.yaml` это уже указано).
- Если XP/лидерборд/комментарии не сохраняются, откройте логи сервиса на Render (вкладка **Logs**) — при старте сервер печатает либо `✅ MongoDB подключена`, либо явную ошибку подключения. Самые частые причины: `MONGODB_URI` не задан/скопирован с опечаткой, в Atlas не разрешён доступ `0.0.0.0/0` в Network Access, или неверный пароль пользователя базы данных.
- Предупреждение антивируса (например, Avira) о "фишинге" на свежесозданном поддомене `github.io` — частое ложное срабатывание на новые/малопосещаемые страницы, к коду отношения не имеет.

## Локальная разработка

`npm run dev` в корне проекта по-прежнему поднимает фронтенд и бэкенд локально на одном origin через Vite-прокси — все правки в Шагах 1–3 выше применяются только к продакшену (`NODE_ENV=production` / `vite build`).

Единственное отличие: теперь backend'у нужен `MONGODB_URI` даже локально (в `backend/.env`, скопированном из `backend/.env.example`) — без него пользователи/XP/лидерборд не будут сохраняться. Проще всего использовать тот же бесплатный кластер MongoDB Atlas из Шага 0 — он одинаково доступен и с локальной машины, и с Render.
